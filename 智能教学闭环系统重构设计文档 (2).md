# 1.3 核心流程设计
## 1.3.1 用户认证流程（JWT + RBAC）
用户提交登录表单 → AuthController.login() → BCrypt验证密码 → 查询角色(user_roles+roles) → JwtUtil生成Token → 存入Redis(TTL:24h) → 返回LoginResponse{token,userInfo,roles}。后续请求携带Authorization: Bearer{token}，经JwtAuthenticationFilter拦截验证后，@PreAuthorize注解校验角色权限。

## 1.3.2 RAG智能问答流程（SSE流式）
学生提问(question+videoId+currentTime) → 构建上下文约束 → Embedding向量化 → ES向量检索TopK=5 → Rerank精排TopN=3 → 构建Prompt(System+Context+Question) → Qwen流式生成 → SSE响应流(chunk→source→done) → 前端EventSource逐字渲染。

## 1.3.3 视频异步解析流程
教师上传视频 → OSS存储 → 更新parse_status=1 → RabbitMQ消息队列 → Consumer消费 → FFmpeg提取关键帧 → 并行调用Qwen-VL视觉分析+ASR语音转写 → 时空对齐生成“视-听-文”三元组 → Embedding向量化 → 存入MySQL+ES → parse_status=2。

## 1.3.4 学情热力图生成流程
聚合学习行为日志 → 滑动窗口切片(30秒) → 统计回看/暂停/提问频次 → 加权计算Score=0.4*Replay+0.3*Pause+0.3*Questions → 超阈值0.7标记难点 → JSON返回ECharts渲染。

## 2.3.1 角色信息表 (roles)
RBAC权限模型的核心，预置ADMIN/TEACHER/STUDENT三种角色。
CREATE TABLE `roles` (
`id` INT NOT NULL AUTO_INCREMENT COMMENT '角色编号',
`role_name` VARCHAR(50) NOT NULL COMMENT '角色名称',
`role_code` VARCHAR(50) NOT NULL COMMENT '角色代码',
`description` VARCHAR(255) DEFAULT NULL COMMENT '角色描述',
PRIMARY KEY (`id`),
UNIQUE KEY `uk_role_code` (`role_code`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='角色信息表';

## 2.3.2 用户信息表 (users)
存储用户身份信息与BCrypt加密密码，role_id外键关联roles表。
CREATE TABLE `users` (
`id` INT NOT NULL AUTO_INCREMENT COMMENT '用户编号',
`username` VARCHAR(50) NOT NULL COMMENT '登录用户名',
`password` VARCHAR(100) NOT NULL COMMENT 'BCrypt加密密码',
`email` VARCHAR(100) DEFAULT NULL COMMENT '邮箱',
`avatar_url` VARCHAR(255) DEFAULT NULL COMMENT '头像OSS路径',
`role_id` INT NOT NULL COMMENT '角色编号',

PRIMARY KEY (`id`), UNIQUE KEY `uk_username` (`username`),

## 2.3.3 用户-角色关联表 (user_roles)
支持多角色的M:N关系，联合主键防止重复关联。
CREATE TABLE `user_roles` (
`user_id` INT NOT NULL, `role_id` INT NOT NULL,
PRIMARY KEY (`user_id`, `role_id`),
CONSTRAINT `fk_ur_user` FOREIGN KEY (`user_id`) REFERENCES `users` (`id`) ON DELETE CASCADE,
CONSTRAINT `fk_ur_role` FOREIGN KEY (`role_id`) REFERENCES `roles` (`id`) ON DELETE CASCADE

# 4.1 设计系统规格 (DESIGN.md)
本设计系统参考 Google Stitch 提出的 DESIGN.md 规范，并融合 GitHub 热门项目 awesome-design-md 的设计令牌体系。核心理念：通过结构化 Markdown 文档定义设计系统，使 AI Agent 能像读取 API 文档一样理解 UI 规范，实现像素级一致的界面输出。前端开发采用 Mock 数据模式，通过 Vite Plugin Mock 或 MSW (Mock Service Worker) 模拟后端接口，实现前后端并行开发。

## 4.1.1 多角色设计风格策略
系统服务三种角色，采用“共享基底 + 角色主题”架构。共享底层包括布局格子、间距系统、组件交互状态、响应式断点等基础规范；角色主题通过 CSS 变量切换颜色、字体、阴影等视觉层。首页根据 JWT Token 中的 roleCode 自动路由到对应角色的主界面。


## 4.1.2 视觉主题与氛围 (Visual Theme & Atmosphere)
学生端（双主题可选）：提供暗色和亮色两套主题，用户可通过顶部切换按钮自由选择，偏好通过 localStorage 持久化。暗色主题参考 Linear 设计理念——深灰蓝背景 (#1a1a2e) 搭配柔和的静藤蓝强调色 (#6366f1)，对比度适中，长时间学习不眼皂。亮色主题参考 Notion 设计理念——纯白背景 + 蓝色强调 (#0075de)，简洁专业，适合白天环境使用。
教师/管理端（浅色主题）：参考 PostHog 设计理念——温暖鼠尾草色调。以暖纸色背景 (#fdfdf8) 为基底，橄榄灰文字 (#4d4f46) 带有黄绿色调，鼠尾草色边框 (#bfc1b7) 创造温暖包围感。整体风格亲切、专业、可信赖，适合数据密集的管理场景。

## 4.1.10 主题切换 CSS 架构
通过 Vue 3 的 composable 函数 useTheme() 根据用户角色自动切换主题。核心逻辑：登录后从 useUserStore 获取 roleCode → STUDENT 加载 dark-theme.css，TEACHER/ADMIN 加载 light-theme.css → 通过 document.documentElement.setAttribute('data-theme', 'dark'/'light') 激活。所有组件通过 var(--color-xxx) 引用令片，无需硬编码颜色值。

# 4.3 关键组件代码逻辑
## 4.3.1 useSSEChat Composable
完整的Vue 3 Composable函数，处理SSE流式响应。核心逻辑：添加用户消息 → 创建AI占位消息(isStreaming=true) → fetch建立SSE连接 → 按\n\n分割事件 → 解析data:行的JSON → 根据type(chunk/source/done/error)分别处理。chunk追加到content，source保存到fragments，done标记完成并记录tokenUsage。错误时追加错误信息并关闭isStreaming。
完整代码请参见随附的composables/useSSEChat.ts文件。

## 4.3.2 useTheme Composable (新增)
多角色主题切换。核心逻辑：从 useUserStore 获取 roleCode → STUDENT 设置 data-theme='dark' 并加载暗色主题 CSS → TEACHER/ADMIN 设置 data-theme='light' 并加载浅色主题 CSS → watch roleCode 变化自动切换。初始化时检测 system prefers-color-scheme 作为备选。

## 4.3.3 useUserStore Pinia Store
JWT Token管理。核心状态：token, userInfo, roles。计算属性：isLoggedIn, roleCode, isTeacher, isStudent, isAdmin。方法：login(username,password)调用API并存储token到localStorage；logout()清除所有状态；fetchUserInfo()获取当前用户信息。

## 4.3.4 Axios拦截器配置
请求拦截器自动注入Bearer Token。响应拦截器统一处理错误：code=200直接返回data，code=401自动登出并跳转登录页，其他错误拒绝Promise。

## 4.3.5 引用溯源高亮组件逻辑
点击引用标签触发路由跳转：/student/learn/:videoId?startTime=240&highlight=5。视频播放器接收参数后自动seek到对应时间点，并高亮对应知识片段大纲。引用标签使用静藤蓝 (#6366f1) 背景 + 白色文字的 pill 样式，悬停时显示蓝色光环。

# 4.5 Mock 数据开发模式
前端采用 Mock 数据模式开发，前后端并行开发。推荐方案：MSW (Mock Service Worker) 拦截网络请求，在浏览器 Service Worker 层返回 Mock 数据。Mock 数据存放在 src/mocks/ 目录，按模块分文件组织：auth.mock.ts、video.mock.ts、qa.mock.ts、analytics.mock.ts。Mock 数据必须与 3.8 节定义的 DTO 字段名、类型完全一致。
开发流程：启动 Vite 开发服务器 → MSW 自动注公 Service Worker → 前端发起请求 → MSW 拦截并返回 Mock 数据 → 前端正常渲染。联调时关闭 MSW，请求转发到真实后端。

## 4.5.3 SSE 流式模拟逻辑
qa.mock.ts 中的 SSE 模拟是最复杂的部分，需要模拟服务端逐字输出的 text/event-stream 格式。核心逻辑：将完整回答文本按 2-5 字分片，每片包裹为 data:{JSON}

 格式的 SSE 事件，通过 ReadableStream 逐步发送，最后发送 source 和 done 事件。
// src/mocks/handlers/qa.handler.ts (SSE模拟核心逻辑)
const mockAnswer = '根据视频内容，这个知识点主要讲解了...';
const mockSources = [
{ fragmentId: 1, startTime: 120.5, endTime: 185.0, relevance: 0.95 },
{ fragmentId: 3, startTime: 240.0, endTime: 310.5, relevance: 0.82 },
];

function createSSEStream(answer: string, sources: any[]) {
const encoder = new TextEncoder();
return new ReadableStream({
async start(controller) {
// 1. 逐字发送 chunk 事件
for (let i = 0; i < answer.length; i += 3) {
const chunk = answer.slice(i, i + 3);
const event = `data:${JSON.stringify({ type:'chunk', content:chunk })}

`;
controller.enqueue(encoder.encode(event));
await new Promise(r => setTimeout(r, 30 + Math.random() * 50));
}
// 2. 发送 source 事件
controller.enqueue(encoder.encode(
`data:${JSON.stringify({ type:'source', fragments:sources })}\n\n`));
// 3. 发送 done 事件
controller.enqueue(encoder.encode(
`data:${JSON.stringify({ type:'done', tokenUsage:256 })}\n\n`));
controller.close();
}
});
}

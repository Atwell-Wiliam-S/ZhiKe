## 1.1 系统总体架构

本系统采用**前后端分离的分层模块化单体架构**（Modular Monolith），在逻辑上划分为五个核心层级。选择模块化单体而非微服务的原因：团队规模适中（3-5人）、业务边界清晰但耦合度尚可控、部署运维成本优先。

![系统总体架构图](images/system-overview-arch.jpg)

### 1.1.1 各层职责

| 层级 | 核心组件 | 职责 |
| --- | --- | --- |
| **用户接入层** | Nginx 1.24 | 反向代理、负载均衡、HTTPS卸载、静态资源缓存、Gzip压缩 |
| **前端展现层** | Vue 3 + TypeScript + Vite | SPA单页应用，Composition API + Pinia状态管理 + Element Plus UI + ECharts可视化 |
| **后端业务逻辑层** | Spring Boot 3.2.0 + Java 17 | 8个功能模块，Spring Security Filter Chain统一认证，模块间通过接口调用通信 |
| **数据处理层** | MySQL 8.0 + Redis 7.0 + ES 8.x + RabbitMQ 3.12 | 业务数据持久化 + 缓存/会话 + 向量检索/全文搜索 + 异步任务队列 |
| **外部服务层** | DashScope SDK 2.x + Aliyun OSS 3.x | Qwen系列大模型调用（VL/ASR/Embedding）+ 视频/文件对象存储 |

### 1.1.2 模块化单体边界策略

后端虽部署为单一 JAR，但内部严格按模块划分包路径，模块间**禁止直接注入对方 Bean**，必须通过定义在 `common` 模块中的接口进行调用。未来如需拆分为微服务，只需将模块独立部署并改为 Feign/RPC 调用即可。

```
com.zhike/
├── common/                  # 通用模块（公共组件、工具类、全局异常处理）
│   ├── config/              # 跨模块配置（CORS、Jackson、MyBatis-Plus）
│   ├── exception/           # GlobalExceptionHandler
│   ├── util/                # JwtUtil, Result, PageResult
│   └── dto/                 # 跨模块共享DTO
├── module_security/         # 安全模块
├── module_video/            # 视频模块
├── module_rag/              # RAG智能问答模块
├── module_analytics/        # 学情分析模块
├── module_teacher/          # 教师模块
├── module_student/          # 学生模块
├── module_admin/            # 管理员模块
└── module_ai_adapter/       # AI服务适配器（DashScope统一封装）
```

## 1.3 核心流程设计

### 1.3.1 用户认证流程（JWT + RBAC）

```
用户登录表单
    ↓
AuthController.login()
    ↓ BCrypt验证密码
查询角色 (users → user_roles → roles)
    ↓
JwtUtil.generateToken(userId, roleCode)
    ↓ 存入Redis (TTL: 24h)
返回 LoginResponse { token, userInfo, roles }
    ↓
后续请求: Authorization: Bearer {token}
    ↓
JwtAuthenticationFilter 拦截验证
    ↓ Redis查Token是否存在
@PreAuthorize 注解校验角色权限
```

**关键设计点：**
- Token 存储在 Redis 中，支持主动失效（踢人、改密、登出）
- Token 有效期 24 小时，支持 `/api/auth/refresh` 无感续期
- RBAC 通过 `user_roles` 关联表支持多角色，`@PreAuthorize("hasRole('TEACHER')")` 注解式鉴权

### 1.3.2 RAG 智能问答流程（SSE 流式）

![RAG数据流架构图](images/rag-dataflow-arch.jpg)

```
学生提问 (question + videoId + currentTime)
    ↓
构建上下文约束 (限定当前视频范围)
    ↓
DashScope Embedding 向量化
    ↓
ES kNN 向量检索 TopK=5
    ↓
Rerank 精排 TopN=3 (基于时间相关性 + 语义相似度)
    ↓
构建 Prompt: [System角色] + [Context参考片段] + [Question]
    ↓
Qwen 流式生成 (逐token输出)
    ↓
SSE 响应流: chunk → source → done
    ↓
前端 EventSource 逐字渲染 + 引用溯源高亮
```

**关键设计点：**
- 上下文约束：仅检索当前视频的 knowledge_fragments，避免跨视频干扰
- Rerank 策略：语义相似度 × 0.7 + 时间邻近度 × 0.3（currentTime附近的片段加权）
- SSE 协议：`text/event-stream`，事件类型为 `chunk`（文本片段）、`source`（引用来源）、`done`（完成）、`error`（错误）
- 答案溯源：每个回答附带 context_fragments（参考片段ID + 时间范围 + 相关度），前端可点击跳转到视频对应时间点

### 1.3.3 视频异步解析流程

```
教师上传视频
    ↓
OSS 存储 (返回 ossUrl)
    ↓ 更新 parse_status = 1 (进行中)
RabbitMQ 消息队列 (video.parse.topic)
    ↓
Consumer 消费
    ↓
FFmpeg 提取关键帧 (每30秒一帧)
    ↓ 并行
    ├── Qwen-VL 视觉分析 (识别公式、图表、文字内容)
    └── ASR 语音转写 (DashScope Paraformer)
    ↓
时空对齐 → 生成"视-听-文"三元组
    ↓
DashScope TextEmbedding 向量化
    ↓
存入 MySQL (knowledge_fragments) + ES (向量索引)
    ↓ 更新 parse_status = 2 (完成)
```

**关键设计点：**
- 状态机：`0(待解析) → 1(进行中) → 2(完成) / 3(失败)`，前端轮询 `/api/video/{id}/status`
- 幂等性：相同 videoId 重复触发解析时，检查 parse_status，非0状态返回 409 状态冲突
- 失败重试：RabbitMQ 死信队列 (DLX) + 最多 3 次重试，超过后标记 parse_status=3 并记录 operation_logs
- 多模态冗余存储：knowledge_fragments 表同时保存融合文本(content_text)、原始ASR文本(asr_text)、关键帧URL(keyframe_url)、视觉标签(visual_tags JSON)，支持溯源

### 1.3.4 学情热力图生成流程

```
聚合学习行为日志 (learning_records + qa_logs)
    ↓
滑动窗口切片 (30秒一个窗口)
    ↓ 统计每个窗口内:
    ├── 回放频次 (Replay)
    ├── 暂停频次 (Pause)
    └── 提问频次 (Questions)
    ↓
加权计算: Score = 0.4×Replay + 0.3×Pause + 0.3×Questions
    ↓
归一化到 [0, 1]
    ↓ 超过阈值 0.7 → 标记为难点 (isDifficulty=true)
JSON 返回前端 → ECharts 热力图渲染
```

| 模块 | 包路径 | 核心职责 | 关键类 | API前缀 | 依赖模块 |
| --- | --- | --- | --- | --- | --- |
| 安全模块 | com.zhike.module_security | JWT认证、RBAC、限流 | JwtUtil, JwtAuthFilter | /api/auth/** | common |
| 视频模块 | com.zhike.module_video | 视频上传、解析调度、知识片段CRUD | VideoAnalysisService | /api/video/** | common, ai_adapter |

| 教师模块 | com.zhike.module_teacher | 课程管理、视频管理、知识片段审核 | TeacherService | /api/teacher/** | common, video |
| 学生模块 | com.zhike.module_student | 课程浏览、学习记录、笔记、心跳上报 | StudentService | /api/student/** | common, video |

### 1.4.1 模块间通信规则

- **禁止循环依赖**：模块间依赖单向流动，`ai_adapter` 和 `common` 不依赖任何业务模块
- **接口隔离**：跨模块调用必须通过定义在 `common` 中的 Java Interface，不直接引用实现类
- **事件驱动**：视频解析等耗时操作通过 RabbitMQ 异步解耦，不使用同步调用
- **统一响应**：所有 API 返回 `Result<T>` 包装类，`{ code, message, data, timestamp }`

      - SPRING_PROFILES_ACTIVE=prod

## 1.7 技术决策记录 (ADR)

### ADR-001: 模块化单体 vs 微服务

| 项目 | 决策 |
| --- | --- |
| **状态** | 已决定 |
| **背景** | 系统需支持8个功能模块，团队3-5人 |
| **决策** | 采用模块化单体架构（Modular Monolith） |
| **理由** | 1) 团队规模不需要微服务的运维复杂度 2) 模块间有共享数据（用户、视频），拆分后需处理分布式事务 3) 单体部署更简单，Docker单镜像即可 4) 包路径严格隔离，未来可无缝拆分为微服务 |
| **后果** | 部署简单但扩展性受限；单点故障风险需通过多实例缓解 |

### ADR-002: Elasticsearch vs 专用向量数据库

| 项目 | 决策 |
| --- | --- |
| **状态** | 已决定 |
| **背景** | 需要 kNN 向量检索 + 全文搜索 |
| **决策** | 使用 Elasticsearch 8.x dense_vector |
| **理由** | 1) ES 8.x 原生支持 kNN 向量搜索，性能满足需求 2) 同时满足全文搜索需求（知识片段关键词检索）3) 减少组件数量，降低运维成本 4) 向量数据量预计 < 100万条，无需专用向量数据库 |
| **后果** | 超大规模（>1000万向量）可能需要迁移至 Milvus |

### ADR-003: SSE vs WebSocket

| 项目 | 决策 |
| --- | --- |
| **状态** | 已决定 |
| **背景** | AI问答需要流式输出 |
| **决策** | 使用 Server-Sent Events (SSE) |
| **理由** | 1) 问答是单向流（服务端→客户端），不需要双向通信 2) SSE 基于HTTP，无需额外协议升级 3) 浏览器原生 EventSource API，前端实现简单 4) 与 Spring Boot SseEmitter 天然集成 |
| **后果** | 不支持客户端主动推送；如需双向通信（如实时协作），需引入WebSocket |

### ADR-004: RabbitMQ vs Kafka

| 项目 | 决策 |
| --- | --- |
| **状态** | 已决定 |
| **背景** | 视频解析等异步任务需要消息队列 |
| **决策** | 使用 RabbitMQ |
| **理由** | 1) 视频解析是任务队列模式（Work Queue），非事件流模式 2) RabbitMQ 支持死信队列 + 消息确认，适合任务重试 3) 管理界面完善，运维友好 4) 消息量级低（日均 < 1000条），无需Kafka的高吞吐 |
| **后果** | 如未来需要事件溯源或大规模日志流，需引入Kafka |

# API 接口规范梳理 - 产品需求文档

## 概述
- **摘要**：根据 API 接口规范，梳理现有后端代码的实现情况，分析各个模块的完成度，识别缺失的功能和需要改进的地方，为后续的开发和测试提供指导。
- **目的**：确保现有代码符合 API 接口规范的要求，识别并弥补功能缺口，提高代码质量和可维护性。
- **目标用户**：后端开发人员、测试人员和项目管理人员。

## 目标
- 梳理现有代码与 API 接口规范的对应关系
- 识别缺失的功能和需要改进的地方
- 提供详细的实现计划，指导后续开发
- 确保所有接口都符合规范要求

## 非目标（范围外）
- 不包括前端代码的分析
- 不包括数据库 schema 的修改
- 不包括第三方服务的集成
- 不包括性能优化和安全审计

## 背景与上下文
- 系统采用前后端分离架构，后端使用 Spring Boot 3.2.0
- API 接口规范已在 `03-API接口规范.md` 中定义
- 现有代码结构已按照模块化设计组织
- 部分模块已经实现了基本功能，但可能存在功能缺口

## 功能需求
- **FR-1**：认证授权接口组 - 实现登录、注册、登出、刷新 Token 和获取用户信息功能
- **FR-2**：视频资源接口组 - 实现视频上传、解析、状态查询和知识片段管理功能
- **FR-3**：RAG 智能问答接口组 - 实现流式问答和问答历史查询功能
- **FR-4**：学情分析接口组 - 实现难点热力图、班级学情概览、学生学习画像和高频困惑词云功能
- **FR-5**：学生接口组 - 实现课程列表、课程详情、视频播放信息、心跳上报、进度保存和笔记管理功能

## 非功能需求
- **NFR-1**：接口响应时间不超过 500ms
- **NFR-2**：支持并发请求，能够处理至少 100 个并发用户
- **NFR-3**：严格遵循 API 规范，包括统一响应格式、错误处理和权限控制
- **NFR-4**：代码结构清晰，遵循 Spring Boot 最佳实践，便于维护和扩展

## 约束
- **技术**：使用 Spring Boot 3.2.0，MyBatis-Plus 3.5.5，MySQL 8.0.35
- **业务**：严格按照 API 规范实现，确保与前端对接顺畅
- **依赖**：依赖现有的认证系统和数据库结构

## 假设
- 数据库表结构已按照设计文档创建完成
- 认证系统已实现，能够正确处理 JWT Token
- OSS 签名 URL 生成功能已实现

## 验收标准

### AC-1：认证授权接口组
- **给定**：用户已注册，拥有有效的用户名和密码
- **当**：发送 POST 请求到 `/api/auth/login`，携带正确的用户名和密码
- **然后**：返回 200 状态码和包含 Token 的响应
- **验证**：`programmatic`

### AC-2：视频资源接口组
- **给定**：教师用户已登录，拥有有效的 Token
- **当**：发送 POST 请求到 `/api/video/upload-signature`，携带文件信息
- **然后**：返回 200 状态码和包含上传签名 URL 的响应
- **验证**：`programmatic`

### AC-3：RAG 智能问答接口组
- **给定**：学生用户已登录，拥有有效的 Token
- **当**：发送 POST 请求到 `/api/qa/chat-stream`，携带问题和视频 ID
- **然后**：返回 SSE 流式响应，包含回答内容和参考来源
- **验证**：`programmatic`

### AC-4：学情分析接口组
- **给定**：教师用户已登录，拥有有效的 Token
- **当**：发送 GET 请求到 `/api/analytics/heatmap`，携带课程 ID 和视频 ID
- **然后**：返回 200 状态码和包含热力图数据的响应
- **验证**：`programmatic`

### AC-5：学生接口组
- **给定**：学生用户已登录，拥有有效的 Token
- **当**：发送 GET 请求到 `/api/student/courses`，携带分页参数
- **然后**：返回 200 状态码和包含课程列表的分页响应
- **验证**：`programmatic`
